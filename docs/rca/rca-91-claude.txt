RCA 91: Claude auth path mismatch investigation
Date: 2026-04-19

Summary
- Blueprint does not have a single hard-coded Claude credential location.
- The effective Claude config/auth directory is computed from process env in `safe-exec.js`:
  - `HOME = process.env.HOME || '/home/blueprint'`
  - `CLAUDE_HOME = process.env.CLAUDE_HOME || join(HOME, '.claude')`
  - See `safe-exec.js:19-21`.
- That means the real credential path depends on the image/runtime env, not on the `/data` paths hard-coded elsewhere.
- The source contains a split-brain setup:
  - `entrypoint.sh` hard-codes setup work under `/data/.claude`
  - the app/server/keepalive/read/resume paths use `safe.CLAUDE_HOME`
  - the main `Dockerfile` sets `HOME=/home/blueprint`
  - `Dockerfile.huggingface` sets `HOME=/data`

1. Where does Claude CLI write credentials after /login? Does HOME=/data affect this?
- Blueprint-launched Claude sessions are created in `safe.tmuxCreateClaude()`, which explicitly exports:
  - `CLAUDE_HOME=<safe.CLAUDE_HOME>`
  - `CLAUDE_CONFIG_DIR=<safe.CLAUDE_HOME>`
  - `HOME=<safe.HOME>`
  - See `safe-exec.js:93-100`.
- So, from Blueprint's point of view, `/login` writes wherever `safe.CLAUDE_HOME` resolves.
- `safe.CLAUDE_HOME` is derived from env at server start (`safe-exec.js:19-21`).
- Therefore:
  - if runtime `HOME=/data`, Blueprint points Claude at `/data/.claude`
  - if runtime `HOME=/home/blueprint` and no `CLAUDE_HOME` override, Blueprint points Claude at `/home/blueprint/.claude`

Important source discrepancy
- Main `Dockerfile` sets:
  - `HOME=/home/blueprint`
  - `BLUEPRINT_DATA=/home/blueprint/.blueprint`
  - `WORKSPACE=/home/blueprint/workspace`
  - See `Dockerfile:53-55`.
- But `entrypoint.sh` assumes `/data` everywhere for Claude state:
  - `/data/.claude`
  - `/data/.blueprint`
  - `/data/workspace`
  - See `entrypoint.sh:23-25`.
- There is also a second image definition, `Dockerfile.huggingface`, which sets `HOME=/data` (`Dockerfile.huggingface:40-42`).

Conclusion for Q1
- Yes, `HOME` absolutely affects the credential location unless `CLAUDE_HOME` is explicitly set.
- In this codebase, the actual write path for `/login` is not always `/data/.claude`; it is whatever `safe.CLAUDE_HOME` resolves to.

2. Could there be multiple credential file locations?
- Yes.
- At least these locations are possible from source:
  - `/home/blueprint/.claude/.credentials.json` when using the main `Dockerfile` with no `CLAUDE_HOME` override
  - `/data/.claude/.credentials.json` when using `Dockerfile.huggingface` or when `CLAUDE_HOME=/data/.claude`
- Additional evidence:
  - `docker-compose.uitest.yml` explicitly sets `CLAUDE_HOME=/data/.claude` and `CLAUDE_CONFIG_DIR=/data/.claude`
- So the repo already contains multiple runtime patterns for Claude home/config.

3. Does the entrypoint or any code copy/overwrite credentials?
- I found no code in the requested files that copies or overwrites `.credentials.json`.
- Credential reads:
  - `routes.js` auth check reads `join(CLAUDE_HOME, '.credentials.json')` (`routes.js:83-105`)
  - `keepalive.js` reads `join(CLAUDE_HOME, '.credentials.json')` (`keepalive.js:24-38`)
- Entry-point writes/configures other Claude files under `/data`, but not `.credentials.json`:
  - creates `/data/.claude/.claude.json` (`entrypoint.sh:30-34`)
  - creates `/data/.claude/settings.json` (`entrypoint.sh:57-60`)
  - writes onboarding state to `/data/.claude/.claude.json` (`entrypoint.sh:79-93`)
  - copies skills/CLAUDE.md into `/data/.claude` (`entrypoint.sh:43-47`, `69-74`)
- However, `entrypoint.sh` also runs `claude mcp add-json --scope user ...` without exporting `CLAUDE_HOME` first (`entrypoint.sh:63-67`).
  - So those `claude` invocations use the ambient env, not the `/data` path hard-coded elsewhere.
  - That is another sign the setup is inconsistent.

Conclusion for Q3
- No direct credential overwrite was found.
- The problem is inconsistent path selection, not a credential file copy-back or overwrite in the source reviewed.

4. Is there a mismatch between where /login writes and where claude --resume reads?
- Within Blueprint's own runtime path, `/login` and `claude --resume` are intended to use the same env-derived location.
- Resume path:
  - `/api/sessions/:sessionId/resume` calls `safe.tmuxCreateClaude(..., ['--resume', sessionId])`
  - See `routes.js:691-723`
  - `safe.tmuxCreateClaude()` exports the same `CLAUDE_HOME`/`HOME` values used for any normal Claude session (`safe-exec.js:93-100`)
- Auth/login probes also use the same env base:
  - `/api/auth/login` calls `safe.claudeExecAsync(['--print', 'test'])` (`routes.js:450-456`)
  - `claudeExecAsync()` runs with `env: { ...process.env, TERM: 'xterm-256color' }` (`safe-exec.js:62-79`)
- `keepalive` and auth status also read from `safe.CLAUDE_HOME`:
  - `keepalive.js:24-38`
  - `routes.js:83-105`

Conclusion for Q4
- There is no evidence in these files that Blueprint writes credentials to one path and resumes from a different path inside the same server process.
- The actual mismatch is higher-level:
  - `entrypoint.sh` hard-codes `/data/.claude`
  - app/runtime logic uses env-derived `safe.CLAUDE_HOME`
  - the main `Dockerfile` and `Dockerfile.huggingface` disagree on `HOME`
- So operators can easily look at `/data/.claude/.credentials.json` while Blueprint is actually using `/home/blueprint/.claude/.credentials.json`, depending on which image/env launched the server.

Most likely root cause from source
- Blueprint has a partial migration to `/data`, but it is incomplete.
- `entrypoint.sh` assumes `/data` as the canonical persistent Claude home.
- The application code does not enforce that; it derives Claude home from env at startup.
- The main `Dockerfile` still sets `HOME=/home/blueprint`, which would make all server-launched Claude commands use `/home/blueprint/.claude`, not `/data/.claude`.
- A separate HF Dockerfile sets `HOME=/data`, so behavior depends on which container variant is actually running.

Practical implication
- If the running container was built from the main `Dockerfile` or otherwise starts with `HOME=/home/blueprint`, then:
  - `/login` updates `/home/blueprint/.claude/.credentials.json`
  - `/data/.claude/.credentials.json` stays stale
  - keepalive/auth/resume also operate on `/home/blueprint/.claude`
  - `/data` persistence is bypassed for Claude auth
- If the running container truly has `HOME=/data`, then the Blueprint source paths line up, and the stale `/data/.claude/.credentials.json` symptom would have to be caused by Claude CLI behavior outside the Blueprint source reviewed here.

Answer in one line
- The source-level bug is inconsistent Claude home selection: `/data/.claude` is hard-coded in entrypoint/setup, but actual CLI/auth/resume behavior follows env-derived `safe.CLAUDE_HOME`, and the repo contains conflicting `HOME` definitions (`/home/blueprint` vs `/data`).
