## PART 2: Industry Standards Review

### 1. Dual-Deployment Patterns (HF Spaces + Local Docker)
- Current: Using separate Dockerfiles (Dockerfile, Dockerfile.huggingface).
- Standard: While common for complex apps, industry best practice is a single multi-stage Dockerfile or a single Dockerfile that uses environment variables (e.g., HF_SPACE_ID) to branch behavior in the entrypoint.
- Recommendation: Converge on a single entrypoint.sh that detects the environment and adjust behavior (e.g., skipping sudo/gosu on HF Spaces).

### 2. Persistent Data at /data
- Current: Generally follows the pattern but many fallbacks still point to /home/blueprint.
- Standard: /data is the correct industry standard for persistent volume mounts in containers.
- Recommendation: Remove ALL hardcoded fallbacks to /home/blueprint and /storage. Every component should assume /data as the root if environment variables are missing.

### 3. Port Conventions
- Current: Internal 3000, HF 7860, Local Mapped 6342.
- Standard: HF Spaces requires 7860. Standardizing on one internal port (3000 or 7860) across all environments simplifies configuration.
- Recommendation: Use PORT=3000 as the default internal port and 7860 only when explicitly set (e.g., by HF).

### 4. HF Spaces Configuration
- Current: YAML frontmatter in README.huggingface.md.
- Standard: HF Spaces look at the main README.md for frontmatter.
- Recommendation: Move the YAML frontmatter to the primary README.md or ensure HF is explicitly told to look at README.huggingface.md (though the former is more standard).

### 5. Non-Standard Setup Fixes
- User Management: The use of 'gosu' in the local Dockerfile is good practice, but the user 'blueprint' vs 'hopper' (historical) should be standardized. 
- Workspace Trust: Claude CLI's trust mechanism is handled via .claude.json. This is currently spread across multiple files; centralizing this logic in session-utils.js or config.js would be cleaner.
- Theme Hardcoding: Dockerfile hardcodes dark theme; this should ideally be a runtime default in config.js rather than a build-time echo.
