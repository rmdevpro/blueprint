--------------------------------------------------------------------------------
A. STANDARD PATTERNS FOR DUAL-TARGET DOCKER APPS (HF Spaces + Local Docker)
--------------------------------------------------------------------------------

INDUSTRY STANDARD APPROACH:
  Single Dockerfile + multi-stage builds or build args to handle platform
  differences. Most apps that target both HF Spaces and local Docker use:

  Option A (recommended): Single Dockerfile with ARGs
    ARG PLATFORM=local
    ENV HOME=${PLATFORM == "hf" ? "/data" : "/home/appuser"}

  Option B: Two Dockerfiles (current Blueprint approach)
    Dockerfile          ← local/Docker host build
    Dockerfile.huggingface  ← HF Spaces build

  Blueprint's two-Dockerfile approach is common and acceptable but creates
  maintenance burden — any change must be made in both files. The Dockerfile
  for local deploy is currently MORE diverged from HF than necessary.

RECOMMENDED STANDARD:
  The HF Spaces Dockerfile is actually the BETTER of the two. The local
  Dockerfile should be updated to use /data paths (matching HF), with
  the bind mount in docker-compose.yml handling the host path mapping.

  The bind mount /mnt/workspace/blueprint:/data means the INSIDE of the
  container should always use /data/* — but the local Dockerfile doesn't
  reflect this. This is the root misalignment.
