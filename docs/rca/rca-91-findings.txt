RCA: Claude CLI 401 "Invalid authentication credentials" after successful /login

Root Cause:
The system suffers from a "split-brain" configuration regarding the Claude CLI configuration directory (CLAUDE_HOME). 

1. Configuration Discrepancy:
   - Dockerfile sets ENV HOME=/home/blueprint.
   - Dockerfile.huggingface sets ENV HOME=/data.
   - entrypoint.sh and entrypoint-hf.sh hardcode CLAUDE="/data/.claude" for initialization tasks (seeding settings, marking onboarding complete).
   - safe-exec.js calculates CLAUDE_HOME as process.env.HOME + "/.claude".

2. The /login Failure:
   - When a user performs /login in the Blueprint terminal (tmux session), the environment variables HOME, CLAUDE_HOME, and CLAUDE_CONFIG_DIR are exported.
   - However, if the Claude CLI (a Node.js application) or any of its dependencies (like `os.homedir()`) resolves the home directory using the system's password database (/etc/passwd) instead of the HOME environment variable, it will use /home/blueprint/.claude regardless of the environment.
   - The blueprint user was created with a home directory of /home/blueprint.
   - Consequently, /login writes the new credentials to /home/blueprint/.claude/.credentials.json.

3. The 401 Error:
   - The Blueprint application (server.js, keepalive.js, routes.js) uses safe.CLAUDE_HOME to check authentication status.
   - If HOME=/data is set in the container environment, safe.CLAUDE_HOME becomes /data/.claude.
   - The application therefore reads credentials from /data/.claude/.credentials.json.
   - Since /login wrote to /home/blueprint/.claude, the file in /data/.claude remains unchanged and contains the old, expired token.
   - Subsequent calls to 'claude --resume' or the auth check return 401 because they are using the old token from the persistent volume mount point (/data).

Verification:
- The user confirmed that /data/.claude/.credentials.json still shows the old token after a "successful" login.
- entrypoint.sh seeds settings to /data/.claude, but if HOME is /home/blueprint, the CLI will not see them, causing onboarding to repeat or permissions prompts to appear.

Recommended Fix:
1. Unify the home directory and persistent data path. 
2. Ensure CLAUDE_CONFIG_DIR and CLAUDE_HOME are explicitly set in process.env within the server and passed to all child processes in safe-exec.js.
3. Update the blueprint user's home directory in /etc/passwd to match the persistent data path (/data) if persistence is required.
