Blueprint RCA-91 Strategy Review Findings

The proposed strategy of unifying HOME, BLUEPRINT_DATA, and WORKSPACE to live under /data is technically sound and directly addresses the "split-brain" root cause. By aligning the HOME environment variable with the user's home directory in /etc/passwd, we ensure that both the Blueprint application and external CLIs (which may use os.homedir() and thus /etc/passwd) agree on the location of configuration and credentials.

Key Findings & Recommendations:

1. Consistency Across Dockerfiles:
   - The fix must be applied to both 'Dockerfile' and 'Dockerfile.huggingface'. 
   - Currently, 'Dockerfile.huggingface' sets HOME=/data but still creates the blueprint user with a home of /home/blueprint. This is a primary source of the bug in HF environments.

2. Impact on Test Infrastructure:
   - A grep search revealed over 80 hardcoded references to /home/blueprint, mostly in the 'tests/' directory (live tests, mock tests, and test runbooks). 
   - These tests will fail if the environment is changed to /data without corresponding updates to the test helpers and docker-compose.test.yml.
   - Recommendation: Update 'tests/helpers/db-query.js', 'tests/live/*.test.js', and 'docker-compose.*test.yml' to use /data paths.

3. Fallback Paths in Source Code:
   - 'routes.js' (L1039, L1054) and 'db.js' (L8) contain fallbacks to '/home/blueprint'. These should be updated to '/data' to maintain consistency if the environment variable is ever missing.
   - 'public/index.html' (L1768, L3133) has a hardcoded UI fallback for the workspace path. This should be updated to '/data/workspace'.

4. Entrypoint Seeding:
   - 'entrypoint.sh' currently seeds only a minimal settings.json if missing. The Dockerfile seeds a more complete version (including theme and onboarding flags).
   - Recommendation: Ensure 'entrypoint.sh' seeds the same complete settings.json as the Dockerfile to ensure a consistent experience when /data is a fresh volume mount.

5. Safe-Exec Environment:
   - 'safe-exec.js' already correctly passes HOME, CLAUDE_HOME, and CLAUDE_CONFIG_DIR to tmux sessions. This mechanism is correct and will benefit from the unified /data path.

Summary of Required Changes:
- Dockerfile & Dockerfile.huggingface:
    - Change useradd to: useradd -m -d /data -s /bin/bash blueprint
    - Set ENV HOME=/data
    - Set ENV BLUEPRINT_DATA=/data/.blueprint
    - Set ENV WORKSPACE=/data/workspace
    - Update WORKDIR to /data/workspace
- Entrypoint: Update seeding logic to ensure full settings.json is present.
- Routes/DB/Sync: Update hardcoded fallbacks from /home/blueprint to /data.
- Tests: Batch update /home/blueprint to /data across all test files.
